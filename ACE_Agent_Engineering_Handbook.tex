\section{摘要}

ACE Agent (\textbf{A}utomated \textbf{C}lustering \textbf{E}xpert) 是一个面向数据科学的自主聚类分析系统，处于 MVP / 早期可用阶段。系统采用 \textbf{Orchestrator-Worker 多智能体架构}，以 LLM 为推理核心，实现从数据画像、意图路由、专家调度、代码生成、安全沙箱执行、自动调试（自愈）到学术报告生成的完整闭环。

本文档以学术级规格记录系统的：
\begin{itemize}[nosep]
    \item 整体架构与模块职责
    \item 通信协议与数据流
    \item 自愈数学模型 (Think-Act-Fix)
    \item 安全沙箱设计
    \item LLM 抽象层与成本治理
    \item 部署与复现路径
\end{itemize}

\medskip
\textbf{运行环境}: Conda \texttt{Tumor\_Subtype\_Agent}, Python 3.10+, Windows 11 Pro.\\
\textbf{代码规模}: $\sim$7,300 行核心代码（含 benchmark）, $\sim$3,300 行测试代码, 230 项测试通过, 覆盖率 $\sim$65\%.\\
\textbf{硬件}: 本地 RTX 4060 Ti 8G (开发/小模型), 远端 NVIDIA A4000 16G (训练/中等模型).

\subsection{架构模式}

ACE Agent 采用 \textbf{Orchestrator-Worker 模式}，三层结构：

\begin{enumerate}[nosep]
    \item \textbf{路由层 (Routing Layer)}: MasterRouter 负责语义意图识别
    \item \textbf{编排层 (Orchestration Layer)}: ACESupervisor 负责全局调度、会话管理、结果聚合
    \item \textbf{执行层 (Execution Layer)}: 7 个领域专家 Agent 并行执行聚类任务，每个专家内含 Think-Act-Fix 自愈循环
\end{enumerate}

\section{核心模块详解}

\subsection{MasterRouter --- 意图路由器}

\textbf{位置}: \texttt{agent\_core/router.py} (85 行)

Router 使用 LLM 对用户输入进行三分类语义识别：

\begin{tcolorbox}[title=三种意图, colback=blue!5]
\begin{description}[nosep, style=nextline]
    \item[NEW\_TASK] 在当前数据集上执行新的聚类实验。触发条件：用户想运行实验、获取分析结果。
    \item[FOLLOW\_UP] 追问已有结果、解释理论、学术咨询。不需要重跑算法。
    \item[CODE\_EXAMPLE] 用户索要代码本身，非执行实验。(P0.5 新增)
\end{description}
\end{tcolorbox}

\textbf{兜底策略}: Router 异常时 fallback 到 FOLLOW\_UP (而非 NEW\_TASK)，避免误触发昂贵的完整分析流程。JSON 解析失败时、未知意图值时均遵循此原则。

\textbf{设计要点}:
\begin{itemize}[nosep]
    \item 强制返回 JSON: \texttt{\{"intent": "...", "target\_dataset": "...", "reasoning": "..."\}}
    \item 正则提取 JSON (容错 LLM 输出的额外文本)
    \item 仅接受已知意图值，未知意图 fallback 到 FOLLOW\_UP
\end{itemize}

\subsection{ACESupervisor --- 全局编排器}

\textbf{位置}: \texttt{agent\_core/supervisor.py} (345 行)

\subsubsection{初始化与专家注册}

\begin{lstlisting}[language=Python]
# 默认激活专家 (P0.5 升级, 2026-05 移除 critic)
_DEFAULT_ACTIVE_EXPERTS = ["centroid", "topology", "zoo"]

def __init__(self):
    self.router = MasterRouter()
    self.knowledge_engine = KnowledgeEngine()
    self.experts = build_expert_registry()  # 6 个活跃专家
    self.memory = []  # 会话记忆
\end{lstlisting}

\textbf{专家注册表} (via \texttt{build\_expert\_registry()}):
\begin{itemize}[nosep]
    \item 活跃 (6): centroid, topology, zoo, critic, dimension, ensemble
    \item critic 为独立后验审计方，ensemble 为共识融合方，均不参与并行池
    \item 2026-05 清理: multi\_view (旧版骨架) 与 deep\_representation (与 dimension 重叠) 已移除
\end{itemize}

\subsubsection{编排主流程}

\begin{enumerate}[nosep]
    \item \textbf{语义路由}: 调用 \texttt{MasterRouter.analyze\_intent()} 得到意图
    \item \textbf{RAG 增强}: 从 ChromaDB 检索相关知识注入用户 prompt
    \item \textbf{意图分流}:
    \begin{itemize}[nosep]
        \item FOLLOW\_UP $\rightarrow$ \texttt{\_handle\_follow\_up()} (纯 LLM 回答)
        \item CODE\_EXAMPLE $\rightarrow$ \texttt{\_handle\_code\_example()} (生成代码 Markdown，不走沙箱)
        \item NEW\_TASK $\rightarrow$ \texttt{\_execute\_full\_analysis()} (执行完整实验)
    \end{itemize}
    \item \textbf{高维感知}: 当 $n\_features > 2$ 时自动激活 dimension 专家
    \item \textbf{并行执行}: 遍历 active\_experts，每个专家调用 \texttt{execute\_with\_self\_correction()}
    \item \textbf{结果排序}: 按 score 降序排列 all\_results
    \item \textbf{LLM 摘要}: 调用 \texttt{summarize\_report()} 生成中文总结
    \item \textbf{LaTeX 报告}: 自动生成学术 PDF (静默失败不阻塞 UI)
\end{enumerate}

\subsubsection{错误报告增强 (P0.5)}

\texttt{\_error\_report()} 汇总每个专家最后 3 行日志作为排错依据，而非仅输出通用错误信息。

\subsection{BaseExpert --- 自愈循环框架}

\textbf{位置}: \texttt{expert\_sub\_agents/base.py} (205 行)

这是 ACE Agent 最核心的机制。所有专家继承 \texttt{BaseExpert} 并实现 \texttt{\_generate\_code()}。

\subsubsection{Think-Act-Fix 数学模型}

定义符号：
\begin{itemize}[nosep]
    \item $C_0$: 初始生成的代码 (LLM 产出)
    \item $\mathcal{S}$: 沙箱执行算子, $\mathcal{S}(C) \rightarrow (success, artifacts, error)$
    \item $\mathcal{F}$: 修复算子, $\mathcal{F}(C, e, i) \rightarrow C_i$, 其中 $e$ 为错误信息, $i$ 为尝试次数
    \item $N_{max} = 3$: 最大重试次数
    \item $\mathcal{A}$: artifacts 验证算子, $\mathcal{A}(r) \equiv r.success \wedge r.artifacts \neq \emptyset$
\end{itemize}

\textbf{算法流程}:

\begin{enumerate}[nosep]
    \item \textbf{Think}: LLM 生成初始代码 $C_0 = \text{expert}.\textit{generate\_code}(client, dataset, prompt)$
    \item \textbf{Act}: 沙箱执行 $r = \mathcal{S}(C_0)$
    \item \textbf{Evaluate}: 检查 $\mathcal{A}(r)$
    \begin{itemize}[nosep]
        \item 若 $\mathcal{A}(r) = true$: 成功, 返回 artifacts 中的 AlgorithmRunResult 列表
        \item 若 $r.success \wedge r.artifacts = \emptyset$: \textbf{软失败} --- 注入 artifacts 约定提示后重试
        \item 若 $\neg r.success$: \textbf{硬失败} --- 注入错误 traceback 后重试
    \end{itemize}
    \item \textbf{Fix}: $C_{i} = \mathcal{F}(C_{i-1}, e_{i-1}, i)$, $i \in [1, N_{max}]$, 跳回步骤 2
    \item 若 $i > N_{max}$: 重试耗尽, 返回空结果
\end{enumerate}

\textbf{软失败检测 (P0.5-Bug3 修复)}: 原实现仅检查 \texttt{success=True}，忽略 \texttt{artifacts=\{\}} 情况。修复后判据为 $\mathcal{A}(r) = success \wedge artifacts \neq \emptyset$。

\textbf{软失败根因诊断}: 检测代码中的常见模式错误：
\begin{itemize}[nosep]
    \item \texttt{if \_\_name\_\_ == "\_\_main\_\_":} 守卫 --- 沙箱中 \texttt{\_\_name\_\_ = "\_\_ace\_sandbox\_\_"}
    \item \texttt{def main() / def run()} 定义但未调用
\end{itemize}

\subsubsection{Artifacts 契约}

所有专家生成的代码必须遵循此约定，写入沙箱注入的 \texttt{artifacts} 字典：

\begin{lstlisting}[language=Python]
artifacts[algo_name] = {
    "labels": <list or np.ndarray>,
    "metrics": {"score": float, "score_source": str, ...},
    "plot_path": str  # matplotlib 保存的图片路径
}
\end{lstlisting}

\textbf{禁止}: 写入其他变量名、将逻辑隐藏在未调用函数中。

\subsubsection{代码清洗}

LLM 经常在代码外包裹 Markdown fence (\texttt{```python ... ```})。\texttt{\_strip\_code\_fences()} 在沙箱执行前移除这些包装。

\subsection{CoderSandbox --- 安全沙箱}

\textbf{位置}: \texttt{tools/coder\_sandbox.py} (340 行)

\subsubsection{安全模型}

三层防护：

\begin{enumerate}[nosep]
    \item \textbf{受限命名空间}: 仅开放 SAFE\_BUILTINS (28 个安全内置函数), 不包含 \texttt{eval/exec/compile/open/\_\_import\_\_} 等危险函数
    \item \textbf{Wall-Clock 超时}: 通过 \texttt{threading.Timer} 实现，默认 $t_{max} = 60s$ (环境变量 \texttt{ACE\_SANDBOX\_TIMEOUT\_SEC} 可配)
    \item \textbf{内存上限}: 基于 \texttt{psutil} 的 delta-RSS 监控，默认 $M_{max} = 2048$ MiB (环境变量 \texttt{ACE\_SANDBOX\_MEMORY\_MB} 可配)
\end{enumerate}

\textbf{超限信号}: 抛出 \texttt{SandboxResourceExceeded(reason, detail)}，其中 \texttt{reason} $\in$ \{"timeout", "memory"\}。自愈循环捕获此异常后不重试，直接上报。

\subsubsection{内存监控实现}

\begin{lstlisting}[language=Python]
class _MemoryWatchdog(threading.Thread):
    """后台线程每 0.5s 轮询当前进程 RSS。

    追踪 RSS 增量 (RSS_current - RSS_baseline) 而非绝对值，
    使得基线进程内存 (numpy, torch) 不计入限额。
    """
    def run(self):
        proc = psutil.Process(os.getpid())
        while not self._stop_event.is_set():
            current_mb = proc.memory_info().rss / (1024 * 1024)
            delta_mb = max(0.0, current_mb - self._baseline_mb)
            if delta_mb > self.limit_mb:
                self.exceeded = True
                return
\end{lstlisting}

\subsubsection{Artifacts 救援机制}

沙箱执行后，若标准 \texttt{artifacts} 字典为空，启动 broad-rescue 扫描：

\begin{enumerate}[nosep]
    \item 检查 \texttt{exec\_env["result"]} 是否符合 artifacts 形状
    \item 扫描所有用户定义变量 (排除注入变量、私有变量、可调用对象)，寻找符合 artifacts 形状的 dict
\end{enumerate}

Artifacts 形状识别 (\texttt{\_looks\_like\_artifacts}): 非空 dict，至少包含一个值为含有 \texttt{"labels"} 键的字典。

\subsection{UniversalLLMClient --- LLM 抽象层}

\textbf{位置}: \texttt{tools/llm\_client.py} (415 行)

\subsubsection{Provider 架构}

\begin{lstlisting}[language=Python]
# Abstract Base Class
class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages, **kwargs) -> str: ...
    def count_tokens(self, text) -> int: ...

# Concrete Implementations
class OpenAICompatibleProvider(LLMProvider): ...  # 通用 OpenAI-compat

\subsubsection{Fallback 策略}

\begin{itemize}[nosep]
    \item 触发条件: HTTP 429 / 500 / 502 / 503 / 504 或请求超时
    \item 次数限制: 最多 1 次 fallback (primary $\rightarrow$ secondary)
    \item 日志: fallback 事件写入 \texttt{outputs/llm\_trace.jsonl}
\end{itemize}

\section{自愈能力的形式化分析}

\subsection{自愈成功率模型}

定义自愈成功率 $H$:

\begin{equation}
H = \frac{|{C_i: \mathcal{A}(\mathcal{S}(C_i)) = true, i \in [0, N_{max}]}|}{|{C_0: \text{generated}}|}
\end{equation}

其中 $C_0$ 为 LLM 生成的初始代码，$C_i (i \geq 1)$ 为第 $i$ 次修复后的代码。

当前系统 $N_{max} = 3$，$H$ 的期望值取决于：
\begin{itemize}[nosep]
    \item $P_{gen}$: 初始生成代码直接成功的概率 (取决于 LLM 能力 + prompt 质量)
    \item $P_{fix}$: 单次修复成功的概率 (取决于错误信息质量 + LLM 纠错能力)
\end{itemize}

\begin{equation}
H = 1 - (1 - P_{gen}) \cdot (1 - P_{fix})^{N_{max}}
\end{equation}

\subsection{修复成功率的影响因子}

\begin{enumerate}[nosep]
    \item \textbf{错误信息质量}: 沙箱提供完整 traceback，帮助 LLM 精确定位
    \item \textbf{软失败诊断}: 检测 artifacts 约定违反模式，给出针对性提示而非通用错误
    \item \textbf{代码约束}: 显式导入要求、artifacts 契约文档、沙箱环境说明
    \item \textbf{重试次数}: $N_{max} = 3$ 是成本与成功率的折中选择
\end{enumerate}

\subsection{成本模型}

设每个专家组 $\mathcal{E}$ 包含 $N$ 个专家并行执行。单次任务的总 Token 消耗:

\begin{equation}
T_{total} = T_{route} + T_{rag} + \sum_{e=1}^{N} \sum_{i=0}^{r_e} (T_{gen}^{(i)} + T_{code}^{(i)})
\end{equation}

其中 $r_e$ 为专家 $e$ 的实际重试次数 (均 $\leq N_{max}$), $T_{route}$ 为 Router 消耗, $T_{rag}$ 为 RAG 查询消耗。

预估成本:
\begin{equation}
\text{Cost}_{USD} = \frac{T_{total}^{input}}{1000} \cdot c_{input} + \frac{T_{total}^{output}}{1000} \cdot c_{output}
\end{equation}

\textbf{风险}: 自愈 3 次 $\times$ N 个专家可能导致 Token 指数膨胀。当前已通过 trace 日志追踪，但策略层面尚无预算上限熔断机制 (Phase 1 规划)。

\subsubsection{2.1 Ensemble Consensus Expert}
\begin{itemize}[nosep]

    \item YlOrRd 共现矩阵热力图 (Web UI)，降采样 max 500$\times$500
    \item 条件触发门控：Critic endorsed + confidence $\ge$ 0.75 时自动跳过 Ensemble

\end{itemize}

\section{已知局限与注意事项}

\subsection{自愈循环}

\begin{itemize}[nosep]
    \item $N_{max} = 3$ 是经验值，对复杂算法 (如自定义 PyTorch 训练) 可能不足
    \item 修复依赖 LLM 能力: 低质量 traceback 可能导致修复失败循环
    \item 无预算熔断: 极端情况下 Token 成本可膨胀至 $(N_{max}+1) \times |experts|$ 倍
\end{itemize}

\subsection{沙箱}

\begin{itemize}[nosep]
    \item 超时由 \texttt{threading.Event.wait()} 实现: worker 线程在 C 扩展中时不可中断
    \item 内存监控基于 \texttt{psutil} 轮询 (每 0.5s), 瞬态峰值可能漏检
    \item Windows 环境不支持 \texttt{resource} 模块, 故未限制 CPU 时间
\end{itemize}

\subsection{RAG 系统}

\begin{itemize}[nosep]
    \item \texttt{knowledge\_engine.py} 仅 102 行, 不支持多领域语料治理
    \item 向量库为本地 ChromaDB 持久化, 多用户场景需独立部署
\end{itemize}

\subsection{多用户与安全}

\begin{itemize}[nosep]
    \item 无多用户/会话隔离: 当前仅限单机 Demo
    \item 无 PII 扫描或数据脱敏: 企业场景需额外措施
    \item \texttt{.env} 文件中的 API Key 需妥善管理, 不可提交至版本控制
\end{itemize}
